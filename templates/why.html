Why Does This Exist? How We Got Here

Background
KrakenMeister's civ builder stopped working

KrakenMeister's civ builder is an awesome tool, but it was built against a frozen snapshot of an older version of the game's data file. Every unit, tech, building, and string in AoE2:DE lives inside a binary file called empires2_x2_p1.dat. KrakenMeister's system baked in assumptions about that file — which ID slots were empty, which ranges of string IDs were safe to use — based on the game as it existed when KrakenMeister was written. DLC updates kept shipping after that, and those assumptions stopped being true.

The South American biome collision

The most visible breakage came from the South American DLC. That update added new environmental units — herdable animals, huntable animals, predators, berry varieties, terrain objects — and the game engine assigned them fresh unit IDs in ranges that happened to overlap with the "spare" slots KrakenMeister relied on for custom unique units. The result: maps that were working fine before the DLC would suddenly spawn the wrong thing entirely. Instead of berries, herdable animals, huntable animals, and predators, you'd get patches of neutral (Gaia) unique units — because that slot now meant something completely different. This isn't a bug in KrakenMeister's logic; it's a structural consequence of working from a static snapshot of the dat that was never updated.

The Solution Presents Itself

I used KrakenMeister's civ builder to build all of my custom civs. My brother and I would use these OP civs to beat up on as many extreme AIs as we could stand. Eventually, I started direct dat file editing. When I wanted to create more elaborate and crazy civs, I employed AI: new unique units, aura effects, multiple technologies and bonuses. After all of this editing I realized - I was already part of the way there.

While it has been 10 months since the last KrakenMeister civ update, the Fritz' fork has been serving the community well, until the previously mentioned South American biome collisions.

What Empire Forge does differently

Empire Forge reads your actual installed copy of the game every time you build. Rather than relying on a frozen snapshot, it loads the real dat file from your machine, scans it live, and uses only what is confirmed empty or safe in your current version of the game. Specifically:

Future-Proof
  • Unit slots are verified by scanning all ~60 vanilla civs in the live dat before anything is written — if a slot isn't actually spare, we don't use it.
  • String IDs are allocated from a pool extracted from your installed game's string file, checked at build time against what's actually occupied, not assumed safe from an old snapshot.
  • Civilizations All civs get read from your local game files; there will never be a civilization mismatch error.
  • Chronicles / Antiquity DLC civs (Achaemenids, Athenians, Spartans, Macedonians, Thracians, Puru) are detected and excluded automatically — they occupy civ slots in a separate pool that would corrupt if overwritten.
