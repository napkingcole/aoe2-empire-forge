# AoE2 DE Mod Builder – Modding Notes & Debug Guide

This document captures what we've learned about the game's `.dat` file, the genieutils library, and the engine's quirks, as a reference for future debugging and development.

---

## Project Overview

`build.py` reads `empires2_x2_p1_clean.dat` (never modified), applies all changes in Python, and writes the result to `empires2_x2_p1.dat`. Language strings are written to all 15 locale files under `ui/resources/`.

The clean DAT was generated by **Krakenmeister's Civilization Builder**, which sets up 12 custom civs in slots 1–12 with their base bonuses, UUs, and UT scaffolding. `build.py` handles everything that Krakenmeister can't: stat overrides, multi-building training, new units, civ bonuses, complex effect chains, etc.

```
# Run the build:
./venv/bin/python3 build.py

# Package for deployment:
./venv/bin/python3 deploy.py [optional_prefix]
```

## Mod Community Created Reference Guide

Inside the directory `AoE2DE_UGC_Guide-main` is the git repository for the **AOE2:DE User Generated Content documents**, outlining known properties, attributes, damage calculations, etc, based on real-world testing and research by experienced modders. Considered the 'modding bible' of the community. Refer to this guide and update this document as needed.

## DAT File Structure

### Top-Level Objects

```python
data = DatFile.parse("empires2_x2_p1_clean.dat")

data.civs[]         # List of 57 Civ objects (0 = template/"gaia")
data.effects[]      # List of Effect objects
data.techs[]        # List of Tech objects
data.unit_headers[] # Shared unit AI/task definitions (indexed by unit ID)
```

### Civs and Units

- `data.civs[0]` is the **template civ** — a "gaia" baseline the engine references for certain lookups. Modifications to individual civs (2–12) are independent.
- `civ.units[id]` — indexed by unit ID. Unit IDs are not sequential enumerations; they're array indices. Unit ID = its position in the array.
- **All civs must have the same number of units.** The engine crashes on map load if any civ has a different unit count. When adding a new unit, append to ALL 57 civs.
- `civ.resources[]` — per-civ resource values (e.g., resource 213 = Keshik gold multiplier).

### Unit Object

Key fields accessed during modding:

```python
unit.id                         # Unit ID (= array index)
unit.enabled                    # 0 = disabled (not trainable), 1 = enabled
unit.hit_points
unit.speed
unit.language_dll_name          # String ID for unit name displayed in UI
unit.type_50.attacks            # List of AttackOrArmor (class_, amount)
unit.type_50.armours            # List of AttackOrArmor (class_, amount)
unit.type_50.blast_damage       # AOE secondary damage multiplier (negative = slow aura)
unit.type_50.blast_attack_level # 0=none, 1=allies, 2=all, 3=resources
unit.type_50.max_range
unit.type_50.displayed_attack
unit.creatable.resource_costs   # List of ResearchResourceCost (type, amount, flag)
unit.creatable.train_locations  # List of train location objects (unit_id=building, button_id, train_time, hot_key_id)
unit.creatable.creatable_type   # 1 = hero (one-at-a-time)
unit.creatable.hero_mode        # 1 = hero
unit.dead_unit_id               # Unit spawned when this unit dies
unit.dying_graphic              # -1 = skip dying animation (instant death)
```

### Effects and EffectCommands

Effects are lists of `EffectCommand` objects. An effect is assigned to a tech; it fires when the tech is researched.

```python
EffectCommand(type=EC_TYPE, a=unit_or_class, b=class_or_building, c=attribute, d=value)
```

**EC Types:**
| Constant | Value | Description |
|----------|-------|-------------|
| EC_SET | 0 | Set attribute c of unit a (or class b) to d |
| EC_RESOURCE | 1 | Modify resource: a=resource_id, b=0(set)/1(add), d=amount |
| EC_ENABLE | 2 | Enable/disable unit a at building b (b=1 enables, b=0 disables) |
| EC_UPGRADE | 3 | Replace all existing instances of unit a with unit b |
| EC_ADD | 4 | Add d to attribute c of unit a / class b |
| EC_MULTIPLY | 5 | Multiply attribute c of unit a / class b by d |
| EC_SPAWN | 7 | Spawn units: a=unit_id, b=building_id, c=count |

**Targeting rules:**
- `a = unit_id, b = -1` → applies to that specific unit only
- `a = -1, b = class_id` → applies to all units of that class (e.g., CLASS_CAVALRY)
- Both can be set simultaneously for class-scoped unit targeting in some commands

**Key attribute IDs (c parameter):**
| Constant | Value | Description |
|----------|-------|-------------|
| ATTR_HP | 0 | Hit points |
| ATTR_LOS | 1 | Line of sight |
| ATTR_SPEED | 5 | Movement speed |
| ATTR_ARMOR | 8 | Armor (encode: class×256 + amount) |
| ATTR_ATTACK | 9 | Attack (encode: class×256 + amount) |
| ATTR_RELOAD_TIME | 10 | Attack reload (lower = faster) |
| ATTR_ACCURACY | 11 | Accuracy percent |
| ATTR_MAX_RANGE | 12 | Maximum attack range |
| ATTR_WORK_RATE | 13 | Building/unit work rate multiplier |
| ATTR_BLAST | 22 | Blast/trample radius |
| ATTR_BONUS_DMG_MOD | 24 | Incoming bonus damage multiplier |
| ATTR_TRAIN_LOC | 42 | Training building (Marauders mechanism — REPLACES) |
| ATTR_DEAD_UNIT | 57 | Unit spawned on death |
| ATTR_MAX_CHARGE | 59 | Charge attack max energy (Comitatenses) |
| ATTR_RECHARGE_RATE | 60 | Charge attack recharge rate |
| ATTR_CHARGE_EVENT | 61 | Charge trigger (0=off, 1=on attack) |
| ATTR_CHARGE_TYPE | 62 | Charge type (1=melee bonus) |
| ATTR_HEAL_RATE | 63 | Passive heal aura rate |
| ATTR_TRAIN_TIME | 101 | Unit creation time |
| ATTR_EXTRA_PROJ_1 | 102 | Extra projectile count part 1 |
| ATTR_EXTRA_PROJ_2 | 107 | Extra projectile count part 2 |
| ATTR_REGEN_HP | 109 | HP regeneration per minute |
| ATTR_UI_REFRESH | 158 | No-op toggle (pair with ATTR_TRAIN_LOC) |

**Encoding attack/armor values:**
```python
# Attack or armor values encode class and amount into a single float:
def encode_armor(armor_class: int, amount: int) -> float:
    return float(armor_class * 256 + amount)

# Example: +3 melee attack
EC_ADD(a=unit_id, b=-1, c=ATTR_ATTACK, d=encode_armor(MELEE_CLASS, 3))
# MELEE_CLASS = 4, PIERCE_CLASS = 3
```

### Techs

```python
tech.civ                 # -1 = all civs, N = only fires for civ N

tech.resource_costs      # Tuple of 3 ResearchResourceCost
tech.repeatable          # 0 = one-shot, 1 = repeatable
```

## Mechanism Patterns

### Adding a New Unit

All 57 civs must get the new unit, or the engine crashes. Typical pattern:

```python
new_id = len(data.civs[0].units)   # next available index

for civ in data.civs:
    src   = civ.units[SOURCE_UNIT_ID]
    clone = _clone_unit(src, data.version)  # serialization round-trip
    clone.id      = new_id
    clone.enabled = 0   # disabled by default; tech or direct assignment enables it
    civ.units.append(clone)

# Apply stat changes to the specific civ only:
unit = data.civs[CIV_TARGET].units[new_id]
unit.hit_points = 95
```

`_clone_unit` does a serialization round-trip (to_bytes → from_bytes) rather than a shallow deepcopy, which is required to avoid shared-reference bugs in genieutils objects.

### Multi-Building Training (Civ Bonus)

To make a unit trainable at multiple buildings from the start, add extra `train_location` entries directly in the DAT:

```python
from copy import deepcopy
extra = deepcopy(unit.creatable.train_locations[0])
extra.unit_id   = BUILDING_STABLE
extra.button_id = 5
unit.creatable.train_locations.append(extra)
```

This works for civ bonuses (permanent). For tech-gated secondary buildings, use the Marauders mechanism below.

### Marauders Mechanism (Tech-Gated Training at New Building)

ATTR_TRAIN_LOC **replaces** the primary training location — it does not add a secondary. To add a secondary building behind a tech:

```python
# In effect:
EC_ENABLE(a=unit_id, b=1, c=-1, d=0.0)       # enable the unit
EC_SET(a=unit_id, b=-1, c=ATTR_UI_REFRESH, d=1.0)
EC_SET(a=unit_id, b=-1, c=ATTR_TRAIN_LOC, d=float(NEW_BUILDING_ID))
EC_SET(a=unit_id, b=-1, c=ATTR_UI_REFRESH, d=0.0)
```

This replaces the unit's training building in the UI, not adds to it. If the unit was at Castle and you do this, it moves to the new building. Use the DAT multi-building approach if you need both buildings simultaneously.

### C-Bonus Age-Scaling Tech Pattern

To apply a civ-specific bonus that scales by age (fires automatically when that civ researches an age):

```python
from copy import deepcopy
TEMPLATE_IDS = {101: 711, 102: 727, 103: 728}  # age_tech_id → template_tech_id

for age_tech, template_id in TEMPLATE_IDS.items():
    new_tech = deepcopy(data.techs[template_id])
    new_tech.civ        = CIV_TARGET
    new_tech.effect_id  = my_effect_id
    new_tech.name       = f"My C-Bonus {age_tech}"
    # required_techs already includes the age tech from the template
    data.techs.append(new_tech)
```

Template techs 711/727/728 have `civ=-1, location_id=-1, research_time=0` — they auto-fire invisibly for the matching age. Setting `civ=CIV_TARGET` restricts the trigger to that civ.

### Charge Attack (Comitatenses Mechanism)

Units that don't natively have charge attacks need all four attributes set:

```python
for uid in CHARGE_UNIT_IDS:
    EC_ADD(a=uid, b=-1, c=ATTR_MAX_CHARGE,    d=5.0)   # charge pool size
    EC_ADD(a=uid, b=-1, c=ATTR_RECHARGE_RATE, d=0.25)  # recharge per second
    EC_SET(a=uid, b=-1, c=ATTR_CHARGE_EVENT,  d=1.0)   # trigger on attack
    EC_SET(a=uid, b=-1, c=ATTR_CHARGE_TYPE,   d=1.0)   # melee charge type
```

### Death-on-Spawn Unit (Monk Explosion)

To create a unit that instantly explodes when spawned:

```python
clone.hit_points              = -1    # negative HP → dies immediately on spawn
clone.type_50.blast_attack_level = 2  # blast hits all units
clone.type_50.attacks[*].amount  = 25 # blast damage

# Gate behind a tech using ATTR_DEAD_UNIT:
EC_SET(a=source_unit, b=-1, c=ATTR_DEAD_UNIT, d=float(clone_id))
```

### Slow-on-Attack Aura (White Feather Guard Pattern)

Set directly on unit in DAT — **not settable via EffectCommand**:

```python
unit.type_50.blast_damage       = -5.0
unit.type_50.blast_attack_level = 2
```

To gate this behind a tech, create a cloned "slow" unit with these fields pre-set (enabled=0), then use `EC_UPGRADE(base → slow_clone)` and `EC_ENABLE(slow_clone, 1)` in the tech effect.

### Death Refund (Tupi Mechanism)

```python
EC_RESOURCE(a=33,  b=0, d=27.0)   # resource 33=27 activates death-refund mode
EC_RESOURCE(a=295, b=0, d=0.33)   # resource 295 = refund rate (0.0–1.0)
```

### Generate-Resources Tasks (Keshik Gold Pattern)

The generate-resources mechanic (action_type=151) is task-based — it is **not** a tech effect attribute. Rules:
- Tasks must exist in BOTH `data.unit_headers[unit_id].task_list` AND in each `civ.units[unit_id].bird.tasks`
- `data.civs[0]` (the template civ) must also have the tasks — the engine checks civ 0 to determine whether a unit CAN run tasks
- The Keshik has 21 separate tasks (one per unit class_id). Copying only 1 task and repeating it 21 times results in gold generation vs only one target class
- Gate via `EC_RESOURCE` that sets the multiplier resource (213 for Keshik) from 0 to a positive value

## Language Strings

Language strings live in `ui/resources/<lang>/key-value-modded-strings-utf8.txt`. The engine reads from these files using the integer IDs stored in DAT fields like `unit.language_dll_name`, `tech.language_dll_name`, etc.

### String ID Conventions

- Unit name: `language_dll_name` field on the unit
- Unit creation button: typically `dll_name + 1000` (but some follow the `6xxx` pattern)
- Tech name: `language_dll_name` on the tech
- Tech button (hover description): `language_dll_description = dll_name + 1000`
- Tech help text (extended tooltip): `language_dll_help = dll_name + 100000`
- Tech tree entry: `language_dll_tech_tree = dll_name + 150000`

Custom strings use IDs in the 79xxx–80xxx range to avoid collisions with vanilla strings.

### Override vs Custom ID

When renaming an existing unit (e.g., Petard → True Believer, Condottiero → Budget Knight), there are two approaches:

**Option A — Override vanilla IDs (recommended for training buttons):**
```python
# In LANGUAGE_STRINGS dict:
5114: "Budget Knight",    # vanilla Condottiero name string
6114: "Budget Knight",    # vanilla Condottiero creation button string
```
This avoids the "empty button" bug where the Barracks/Castle button shows blank when `language_dll_name` points to a custom ID that doesn't exist in the game's base string table.

**Option B — Change the DAT pointer (works for new units):**
```python
unit.language_dll_name = MY_CUSTOM_STRING_ID
```
Works reliably for new units created by the mod (no vanilla button to worry about). For repurposed existing units, Option A is safer.

## Hotkeys

### How Training Button Hotkeys Work

Each `train_location` has two hotkey-related fields that serve different purposes:

- **`tl.hot_key_id`** — the actual hotkey binding used in the building UI. This is what makes the key work. It references a slot ID from the game's hotkey config (`key-value-strings-utf8.txt`).
- **`unit.language_dll_hotkey_text`** — string ID for the key hint shown in the button tooltip. Display only; does not affect which key fires.

**Critical:** when you move a unit to a new building or button slot, you must explicitly set `hot_key_id` to match. Changing only `button_id` and `unit_id` does not update the key binding — it inherits whatever was set before, which may belong to the wrong building/slot.

### Reference hot_key_id Values

Find the right `hot_key_id` by inspecting a vanilla unit that already occupies that slot:

| Slot | `hot_key_id` | Example unit |
|---|---|---|
| Castle btn1 (UU, Q) | 16101 | Iron Pagoda, Konnik, Keshik |
| Castle btn1 (Q, alt) | 16108 | Mangudai, Budget Knight |
| Krepost btn1 (Q) | 16101 | Konnik, Keshik, Huskarl |
| Stable btn3 (E) | 16416 | Heavy Cavalry line |
| Barracks btn2 (W) | 16078 | Pikeman/Spear line |
| Barracks btn3 (E) | 16679 | Condottiero (vanilla); Huskarl after Anarchy |
| Barracks btn4 (R) | 16748 | Huskarl (vanilla), Swole Militia |

**Fixed keys for non-standard slots:** When no vanilla unit occupies the target slot, use a building construction hotkey ID — the first character of its string becomes a fixed key that doesn't change with player config. Example: `hot_key_id=2012` → string starts with G → fixed G key. Use this for Castle/Barracks btn10 and similar slots with no vanilla analogue (e.g. Gainlord, Grenadier).

The `language_dll_hotkey_text` for the matching slot follows the pattern `hot_key_id + 139000` for cavalry-class units (e.g. Stable btn3: `16416 + 139000 = 155416`). This pattern doesn't hold universally — always verify against a known-working vanilla unit.

### Placeholder Slots and EC_SET c=42

When using the Anarchy/placeholder pattern (secondary slot with `unit_id=-1`, revealed by EC_SET c=42), the `hot_key_id` and `button_id` baked into the placeholder are what the game uses after the tech fires. Set them to match the target building slot at DAT-build time.

## Engine Quirks

### ATTR_TRAIN_LOC Replaces, Not Adds

`EC_SET c=42 d=building_id` **replaces** the unit's primary training building. If the unit was trainable at Castle and you do this with Barracks, it moves to Barracks only — it is not available at both. Use the DAT multi-training-location approach for permanent multi-building, or accept that the tech moves it.

### Unit Array Length Must Be Uniform

The engine crashes if any civ has a different number of units than another. Always append new units to ALL 57 civs, not just the target civ. Apply stat changes only to the specific civ after the append loop.

### blast_damage Is Not a Tech Effect Attribute

`unit.type_50.blast_damage` has no corresponding EffectCommand attribute ID. It cannot be changed by researching a tech. The only way to gate it behind a tech is the EC_UPGRADE-to-clone pattern.

### ATTR_WORK_RATE Does Not Affect Generate-Resources Tasks

`EC_MULTIPLY c=13 d=2.0` on a building will speed up unit production, but does NOT affect `action_type=151` (generate resources) tasks. Those only respond to direct mutation of `task.work_value_1` in the DAT, or the multiplier resource (e.g., resource 213).

### Required Tech Count

`required_tech_count` must match the number of non-(-1) entries in `required_techs`. If the count is wrong, the tech may fire at the wrong time or never fire. Always set both together.

### EC_ENABLE(unit, 0) Is Civ-Scoped

`EC_ENABLE(unit_id, 0)` (disable) in a tech effect only disables that unit for the player who researches the tech. Other civs are unaffected. This is useful for replacing base units with upgraded clones after a tech is researched.

### Halberdier Tech Re-Enables Unit 359

The vanilla Halberdier research tech fires `EC_ENABLE(359, 1)`. If Improvised Weapons does `EC_ENABLE(359, 0)` to redirect to the slow clone, researching Halberdier afterward will re-enable unit 359. Accepted tradeoff for the meme mod context.

### Chieftains (Kill-Gold) Only Triggers vs Specific Targets

Resource 274/33 kill-gold only triggers when killing Villagers, Trade Units, or Monks — hardcoded in the engine. It does NOT apply to combat units.

### Charge Attack Requires All Four Attributes

Units without native charge attacks (charge_event=0) require all four EC_ADD/EC_SET calls (max_charge, recharge_rate, charge_event, charge_type). Missing any one of them results in no visible charge behavior.

### Appending to data.techs and data.effects Is Safe

The DAT format is count-prefixed — the game reads exactly N items based on the count at the start of the section. Appending to these lists correctly increments the count during serialization. This has been confirmed working for both effects (via upsert_effect) and techs (Elite Budget Knight upgrade tech, Stormriders age-bonus techs).

## Debugging Tips

**Finding something in the DAT:**
```python
dat = DatFile.parse("...clean.dat")

# Find unit by name fragment:
for uid, u in enumerate(dat.civs[0].units):
    if u and u.name and 'CONDO' in u.name.upper():
        print(uid, u.name, u.hit_points)

# Find tech by name fragment:
for i, t in enumerate(dat.techs):
    if t and t.name and 'condot' in t.name.lower():
        print(i, t.name)

# Find effects using a specific attribute (c=N):
for i, eff in enumerate(dat.effects):
    if eff and eff.effect_commands:
        for ec in eff.effect_commands:
            if ec.c == 42:  # ATTR_TRAIN_LOC
                print(f"effect {i} ({eff.name}): {ec}")
```

**Verifying a change after build:**
Read `empires2_x2_p1.dat` (output), not the clean copy:
```python
dat = DatFile.parse("data/resources/_common/dat/empires2_x2_p1.dat")
u = dat.civs[CIV_BUDGET].units[882]
print(u.hit_points, [(c.type, c.amount) for c in u.creatable.resource_costs])
```

**Build is idempotent:** Running `build.py` multiple times is safe — `upsert_effect` finds effects by name and replaces them rather than appending duplicates. Unit appends (new clones) happen from the clean DAT each time.

**If you get an engine crash on map load:** Most likely a unit array length mismatch. Verify that every civ has the same `len(civ.units)` after modifications.

**If a tech doesn't appear in the UI:** Check `research_locations[0].location_id` and `button_id`. If `location_id=-1`, the tech is invisible (auto-fires). The `civ` field must be `-1` for all-civs or the specific civ index.

**If a unit button is blank/empty:** The `language_dll_name` pointer on the unit may point to a custom string ID that's in the modded language file but not in the game's base string table. Override the original vanilla string ID instead (e.g., 5114 for Condottiero) rather than changing the DAT pointer.
